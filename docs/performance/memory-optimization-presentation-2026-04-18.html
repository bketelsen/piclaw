Executive summary
The biggest issue was not normal cold start. It was hydrating a pathological persisted web:default session packed with oversized inline toolResult payloads. Once that root cause was contained, the rest of the work focused on slimming session bootstrap, delaying heavyweight extensions, and reducing retained main-session state in the live runtime.
  • OOM root cause fixed via session rotation + persistence sanitization
  • Cold path slimmed through gating, lazy loading, and Azure splits
  • Live RSS lowered through warmup reduction, cache caps, cheaper prewarm, and earlier trimming
  • Telemetry improved with startup snapshots and cross-platform RSS visibility

1. What broke, and why it mattered

The first breakthrough was proving that the restart loop was driven by pathological persisted state rather than “ordinary startup cost”.

Root cause: pathological archived session

The archived web:default session that triggered the restart loop was ~95.2 MB, with very large inline tool results and image reads persisted directly into JSONL transcript rows.

  • 18,545 lines total
  • 61.7 MB of persisted toolResult payloads
  • 42.8 MB from toolResult:read alone
  • 27.2 MB from only 18 inline image-read rows
  • Historical failed restarts peaked around 475–477 MB RSS-equivalent
Hydration case Elapsed RSS delta Heap delta External delta VmHWM
Old archived web:default ~638 ms 409.6 MB 201.1 MB 175.2 MB 568.4 MB
Trimmed current session ~248 ms 28.6 MB 13.9 MB 4.6 MB 190.6 MB

Conclusion: the crash loop was caused by hydrating huge persisted payloads, not by normal session creation.

2. The four optimization pillars

Once the root cause was pinned down, the work split naturally into four mutually reinforcing tranches.
Pillar 1

Contain pathological state

Rotate the oversized session, enable automatic rotation, sanitize oversized persisted tool results at load time and append time, and capture clean startup snapshots before further chat activity.

Pillar 2

Slim the session cold path

Move blocking work off session_start, gate extensions by platform/channel, lazy-load heavy Office/CDP/viewer code, and split Azure provider logic from image-generation code.

Pillar 3

Cut retained live memory

Reduce warmup fan-out, cap cached main sessions at two, make recent-chat prewarm lightweight, and lower the pressure threshold so trimming happens before the VM gets close to the historical OOM band.

Pillar 4

Measure continuously

Use /agent/system-metrics, startup-memory snapshots, controlled staged testing, and reproducible history/chart updates so each tranche is evidence-driven rather than intuitive.

3. Optimization timeline

A condensed chronology of what we changed and what each tranche bought us.
Phase A · Immediate containment

Session rotation + persistence sanitization

Rotated the pathological web:default session, enabled automatic session rotation, bounded oversized tool-result persistence, and added clean post-start snapshots.

  • Operational default pushed toward sessionMaxSizeMb=32
  • Oversized image/blob payloads no longer re-accumulate inline
Phase B · Cold-path decluttering

Remove unnecessary work from session bootstrap

Disabled startup warmup by default, removed active workspace-local extensions from the hot path, and moved workspace indexing out of blocking session_start.

  • Real workspace benchmark dropped from 630 ms / 133.3 MB toward the low-300 ms / ~80 MB range
  • OOM loop was no longer reproducible via ordinary restart
Phase C · Extension/path gating

Bundle gating + lazy heavy imports

Gated bundled extensions by platform/channel, lazy-loaded Office and CDP modules, and replaced heavy web viewer session-load paths with lightweight shims plus lazy route registration.

  • Non-web chats dropped from 46 → 44 tools
  • Cold web benchmark reached 215 ms / 25.3 MB after lazy Office/CDP work
Phase D · Azure cold-path split

Move Azure provider work off ordinary session creation

Shifted Azure provider bootstrap to process bootstrap, added a lightweight session shim, and then split image helpers into a separate module so normal chat sessions stop paying that import cost.

  • 169 ms / 26.9 MB after provider-bootstrap + shim split
  • 175 ms / 26.3 MB after image-helper split
Phase E · Live runtime RAM work

Warmup reduction + cache cap + cheaper prewarm

The 2026-04-18 tranche attacked retained main-session state directly with startup warmup controls, a main-session cap/LRU path, lightweight recent prewarm, earlier pressure trimming, and improved RSS visibility in the HUD.

  • Hot-session budget settles at 2 cached main sessions instead of ~6
  • Cross-platform HUD now shows RSS consistently
Phase F · Post-reload verification

Replace checkpoints until the process settles

Repeated post-restart measurements were used to replace the leading checkpoint until the service fell closer to the new baseline, instead of locking in a noisier interactive sample too early.

  • Latest fresh post-start baseline: 212.7 MB RSS
  • Latest light-load live sample: 221.5 MB RSS / 220.4 MB PSS

5. What the latest runtime tranche changed

The 2026-04-18 RAM tranche was the first one targeted primarily at retained live memory rather than cold-start composition.
Change Why Outcome
Startup warmup env controls Keep aggressive recent-chat prewarm disabled or explicitly tunable. Less automatic runtime materialization at startup.
Main-session cache cap / LRU path Stop the hot path from carrying too many main sessions resident. Settles around 2 cached main sessions instead of the previous ~6-shape.
Lightweight recent-chat prewarm Warm directories and resolution caches without spinning full runtimes. Prewarm becomes materially cheaper than full session creation.
Lower pressure threshold (512 → 384 MB) Trim earlier on the crash-prone VM instead of waiting until the process is too close to the historical danger band. Pressure-aware trimming activates sooner.
Cross-platform RSS HUD Make RSS visible on Windows/macOS too; Linux still prefers VmRSS when available. Operational telemetry is more comparable across environments.
